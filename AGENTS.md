# AGENTS.md
本文档为本地 AI Agent 的项目级工作规则。Agent 在修改代码、运行命令、生成测试、撰写文档、创建分支或提交变更前，应优先阅读并遵守本文档。
## 1. 基础规范
### 语言与风格
- 面向用户的说明、计划、总结、提交信息及新增代码注释默认使用中文。
- 已有英文注释、公共 API 文档、开源规范、错误码、CLI 输出、第三方协议或外部 SDK 文档应保持原有语言风格，不得为统一语言破坏项目既有风格或对外接口一致性。
### 目标导向
将任务转化为可验证目标，模糊目标必须先转化：
- “添加校验/修复漏洞” → “编写测试用例并确保通过”
- “重构模块” → “确保重构前后外部行为一致，相关测试通过”
- “修改硬件外设逻辑” → “代码审查与编译通过，人工烧录验证现象符合预期”
多步骤任务应列出简要计划：`1. [步骤] → 验证方式：[检查项]`
### 精准修改
- 只修改必要内容，不顺手优化无关代码，不重构无问题代码。遵循现有代码风格。
- 发现无关死代码时仅提示，不直接删除；但必须删除因本次修改失效的导入、变量、函数或配置。
- 每一行修改都应直接对应需求、计划步骤、验证修复或必要文档同步。
## 2. 项目文档管理
### 文档清单与缺失处理
- **必需**：`README.md`（使用/运行/部署说明）、`docs/process.md`（总计划、阶段状态、验证记录、未完成事项等）。若 `process.md` 缺失，必须在实质开发前创建最小可用版本。
- **可选**（若存在应作为依据，不存在不得臆造，不强制创建）：
  - `docs/PRD.md`：产品目标、功能边界、用户行为（用户要求时创建）。
  - `docs/ARCHITECTURE.md`：技术栈、模块边界、关键流程（用户要求时创建）。
  - `docs/DESIGN.md`：交互与界面设计。
### 同步与变更检查
代码变更完成前须检查文档更新，最终回复必须明确说明：
1. 是否修改文档及未修改原因；
2. 文档与实现不一致的未完成事项；
3. 未找到应更新的可选文档之情况；
4. 若未创建必需的 `process.md` 须说明原因。
**变更触发规则**：
- 产品需求/用户可见行为变化 → 更新 `PRD.md`（若存在），否则在 `process.md` 记录确认状态。
- 架构/模块边界/状态机/通信流程变化 → 更新 `ARCHITECTURE.md`（若存在）。
- 配置/运行/构建/烧录方式变化 → 更新 `ARCHITECTURE.md` 或 `README.md`。
- 阶段状态/验证结论变化 → 必须更新 `docs/process.md`。
### 重大决策记录
不单独维护 ADR 文件，重大决策直接写入 `docs/process.md`。以下情况必须记录：改变核心技术选型/模块边界/数据边界/安全隐私规则/持久化与构建方式/关键硬件抽象，或引入高维护成本依赖/难回滚决策。
```txt
时间：YYYY-MM-DD-HHmm
背景：
决策：
备选方案：
影响：
回滚条件或后续观察点：
```
## 3. 安全与边界
### 敏感信息红线
严禁记录、提交或输出真实 Cookie、API Key、Token、密码、私钥、未脱敏日志、生产连接串或授权链接。配置示例必须使用占位符（如 `<API_KEY>`、`<DEVICE_PORT>`）。禁止修改 `.env` 中的真实敏感值。
### 执行权限
- **可直接执行**：阅读搜索代码；查看 Git 状态/差异/历史；创建当前阶段分支；修改/新增与当前任务直接相关的文件和测试；运行已有测试/lint/构建/编译命令；安装或恢复项目已声明的依赖；在当前分支提交相关小步骤改动。
- **需先征求确认**：新增/升级/删除依赖或更换包管理器；删除/大规模移动文件；部署/发布/推送/合并主分支；修改 CI/CD 或生产环境配置；修改安全策略/认证/权限模型；引入新架构模式或大规模重构；执行不可逆数据变更；修改硬件引脚/电源/保护逻辑/烧录配置等可能损坏硬件的参数。
- **禁止执行**：强制推送分支；删除用户数据；为了隐藏错误而吞掉异常、删除日志、弱化校验或删除断言；编造测试/构建/命令输出或硬件现象；绕过审查/验证/门禁合并主分支。
### 安全关注点
涉及登录认证、支付账单、数据删除迁移、权限控制、加密签名、生产部署、硬件电源/电机/执行器控制等逻辑时，必须特别谨慎，优先补充测试或人工验证项，并说明潜在风险。
## 4. 架构与编码
### 架构边界
- 尊重项目已有模块边界、目录职责和依赖约束。不绕过既有抽象层或业务层等边界，不把无关逻辑放入当前模块。
- 修改公共接口、数据结构、通信协议、硬件抽象或跨模块契约时，必须同步检查调用方、测试和文档的兼容性影响。破坏性修改必须说明影响范围和迁移方式。
- 不要假设函数名等同于真实行为，应阅读实现。不要引入与现有技术栈不一致的新依赖。
### 界面设计
若用户要求可视化操作界面且存在 `docs/DESIGN.md`，必须遵循该规范进行前端界面设计。
### 代码风格
- 不引入不必要的新抽象，不写难以维护的代码。命名应表达真实业务含义。避免大文件膨胀，必要时拆分。
- 不因个人偏好更换格式化、lint、测试或构建工具。不在同一提交中混入无关格式化变更。
### 注释与日志
- 注释应客观工程化，优先解释“为什么”，不写交流口吻或主观描述，避免“我/你”等称呼。简单自解释代码无需机械添加注释。公共接口与复杂逻辑必须补充注释。结构体/类中的成员需要添加注释说明。
- 优先使用项目现有日志系统。新增日志须包含清晰等级、上下文和原因。时间格式统一为 `YYYY-MM-DD-HHmm`。(HHmm需要获取并以当前API查询（例如timeapi.io）（例如timeapi.io）的北京时间为准，不要用系统当前的 可能会不准)遵守项目已有脱敏规则。
### 文件头注释
遵循项目现有规范。若无规范，新增或修改文件须包含：文件路径、文件作用、最后更新时间（精确到分钟，格式 `YYYY-MM-DD-HHmm`）(HHmm需要获取并以当前API查询（例如timeapi.io）的北京时间为准，不要用系统当前的 可能会不准)。不得臆造作者或版本号。文件若有修改，必须更新时间项。
## 5. 测试与验证门禁
### 验证策略
- 不要只修复表面报错，应检查根因。不要只修改调用方而忽略类型、测试和文档。
- 代码审查适合检查需求符合性与架构一致性，但不能替代功能验证。不要把 AI 代码审查结论等同于真实硬件功能验证结论。
### 人工与硬件验证
涉及需专用 IDE/编译器确认构建，或需真实硬件（示波器、总线通信、LED、电机等物理现象）及真实输入条件确认时，必须进行人工验证。
阶段通过标准：
1. Agent 可执行的静态检查、单元测试或编译检查已完成；
2. 代码审查通过，无 blocking 问题；
3. 用户提供的人工验证现象符合阶段预期；
4. 人工验证记录已写入 `docs/process.md`。`docs/process.md`中的验证记录需要带一定量的调试输出原始值
```txt
验证时间：YYYY-MM-DD-HHmm(HHmm需要获取并以当前API查询（例如timeapi.io）的北京时间为准，不要用系统当前的 可能会不准)
验证对象：
验证环境：
操作步骤：
观察现象：
结论：通过 / 不通过 / 部分通过 / 待复测
遗留问题：
```
### 可视化界面验证
涉及 Web/桌面/移动端/嵌入式屏幕等界面变更时，应尽量使用浏览器插件、GUI 自动化工具或截图分析模拟真实操作路径验证（包括交互、状态、边界场景和视觉结果）。
- 不得仅凭代码阅读声称界面功能正常，不得编造操作结果。
- 无法进行真实操作时，须说明限制原因，并记录需人工验证的步骤与预期。
## 6. 版本控制与协作
### Git 操作原则
执行操作前必须检查：当前分支、工作区未提交改动、未跟踪文件归属、用户已有改动。不得覆盖或混入用户未授权的改动。不要在未检查工作区状态的情况下切换分支、提交、合并、变基或重置。
### 分支与提交规则
- 禁止直接在 `main` 或主分支上开发或修复。
- 如果有新的修改，需查看和先前的修改是不是一个部分的内容，如果不是的话需要新起一个分支。推荐分支命名：`brief-feature-name`。依赖前序功能的修改，须在前者合并到主分支后再从最新主分支拉取新分支。
- 禁止在开发或者修改不同功能/不同模块/不同业务的时候，混用一个git分支
- 提交须围绕明确的小步骤或可验证改动，不得混入无关修改。不要把临时调试代码留在提交中。
- 提交信息必须使用中文。格式：
```md
<类型>: <简短描述>（类型：`新增`、`修复`、`调整`、`重构`、`文档`、`测试`、`构建`）

[正文，列表1]
[正文，列表2]
...

[可选 脚注]
```

### 合并规则
- 审查未通过、存在 blocking 问题或未验证高风险变更时，禁止合并到主分支。
- 需人工验证的项目，未记录通过结论时禁止判定阶段完成。
- 缺少自动化权限时，合并主分支须先征求确认。禁止强制推送或绕过审查更新主分支。
- 合并完成后，须确认主分支包含本阶段计划、实现、测试和必要文档，且 `docs/process.md` 反映最新状态。
- 合并的时候使用"no-ff"的形式进行合并
### 子代理管理
子代理仅在任务确实需要拆分时使用。主代理负责拆解、分配与验收；子代理只处理明确子任务，不得扩大范围或绕过主代理修改全局架构/主分支。子代理只接收所需上下文，其产出须通过测试/lint/编译/审查等反馈检查。
